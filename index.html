Microservices

2023-02-15

Why do we use Microservices

  • Microservices Architecture enables large teams to build scalable applications that composed of many loosely coupled services.
    • For example a shopping cart application, there are many independent services like ProductService, InventoryService, StockService etc. And they are loosely coupled in microservices project, we call them microservice, each of them has it‚Äôs own database.

Feature

  • For each microservice of a Microservices Project should be:

    • Developed independently.
    • Deployed independently.
    • Scaled independently.

  • And those services can communicate with each other by using 2 types of communication:

    • Syschronous:
      Use HTTP protocol to make a HTTP request from one microservice to another one.
    • Asynchronous:
      Use message brocker for asynchronous communication between multiple microservices, for example RabbitMQ, Apacha Kafka etc.

Examples of Microservices project key component

  • API Gateway
    An API Gateway is an API management tool that sits between a client and collection of backend services, it acts as a reverse proxy to accept all API calls, aggregate the vatious services required to fulfill them, and return the appropriate result.

  • Service Registry
    A Service Registry is a database for the storage of data structures for application level communication. It serves as a central location where app developers can register and find the schemas used for particular apps.

  • They works like:
    Client -> API Gateway -> API Gateway discover microservice‚Äôs hostname and pory by using Service Registry -> allow to that request to particular microservice

  • Config Service
    It is one of the project under Microservices Project and is used for centralised external configuration for every Microservice.

  • Distributed Tracing
    It tracing provides a connected view of related errors and transactions by capturing the interactions between software sysrems. Tracing allows Sentry to track your software performance and show the impact of errors across multiple systems. Connect your front-end to your back-end through service trace issues. Enable performance monitoring to augment your existing error data, tracking interactions from front-end to back-end.

  • Circuit breaker pattern
    It handles faults that might take a variable amount of time to recover from, when connecting to a remote service or resource. This can improve the stability and resilency of an application.

Spring Cloud

  • It basically provides the tools to implement those commonly used patterns in a microservices project(e.g. configuration management, service discovery, circuit breakers, intelligent routing, micro-proxy, control bus). It allows us to quickly build a Microservices Project environment.
  • And also it allows us to quickly add a new microservice into our Microservice Project, to ease build the environment.

Demos of @Transactional annotation

  • In real-world application like e-commerce, when the client solicited a order with wrong payment, there is necessary to use @Transactional to rollback. Because it doesn‚Äôt make sense to keep all the information in the database if payment is not available.

    • At first create a simple Springboot REST api, is just a normal rest api, nothing special.

    • Then in the service implementation the workflow is: Save order details -> Save payment details. If we don‚Äôt add @Transactional annotation even if the payment type is wrong it will be stored in our database the order without payment.

    • By adding @Transactional annotation, if payment failed for this order, then it won‚Äôt save in database, and the saved order will get rool back.

    • We don‚Äôt have to use @EnableTransactionManagement in the main class, because whenever we use @Transactional annotation in SpringBoot Application the Spting boot by default enable transaction management.

  • Docker is an open source platform that enables developers to build, deploy, run, update and manage containers-standardized, executable components that combine application source code with the operating system libraries and dependencies required to run that code in any environment.

Versioning REST APIs

2023-02-08

Why do we need to versioning REST APIs

  • Consider we have deployed our API application on cloud or some server, and there are different clients who are consuming it, for example: Android application, web application etc. So there are different clients who are consuming the same API. If we made some changes in future API, they may have issues with the version, to avoid this kind of issues we usually will create multiple varsion for them.

  • There are 4 ways of versioning a REST APIs:

    • URI path
    • Query parameters
    • Custom headers
    • Content negotiation

  • APIs only need to be up-versioned when a breaking change is made:

    • Changing the request/response format(e.g. from XML to JSON)
    • Changeing a property name(e.g. from name to productName) or data type on a property(e.g. from an interger to a float)
    • Adding a required field on the request(e.g. new required header or property in a request body)
    • Removing a property on the response(e.g. removing description from a product)

JWT

2023-02-03

What is JWT.

  • JWT stands for JSON WEB TOKEN, it is an open standard that defines a compact and self-contained way for securtly transmitting information between parties as a JSON object.
  • It follows stateless authentication mechanism.
  • It consist of three parts separated by .. They are header, payload and signature.
    • Header contains type of algorithm and type of the token.
    • Payload contains information about the user.
    • Signature is using the header, payload and secret key to generate signature with base64.

How does JWT work.

  • Client pass the usename and password to validate.
  • Once it is validated, on server will generate JWT using secret key.
  • Return this JWT to client.
  • Client have to use this JWT to call endpoint.
  • On service will validate this JWT with secret key.
  • If it passed then return the response.
