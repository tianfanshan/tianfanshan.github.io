FluentAssertions

2023-04-16

Why do we use FluentAssertion

FluentAssertions is a popular testing library in .NET Framework that provides a more readable and expressive way of writing assertions in unit tests. It allows developers to write assertions in a more natural language, which makes it easier to read and understand the intent of the test.
Some benefits of using FluentAssertions in .NET Framework include:

  1. Improved readability: FluentAssertions provides a fluent syntax that reads like natural language, which makes it easier to read and understand the intent of the test.
  2. Better error messages: When an assertion fails, FluentAssertions provides detailed error messages that help developers quickly identify the cause of the failure.
  3. Improved maintainability: The fluent syntax used by FluentAssertions makes it easier to modify tests as requirements change, without having to re-write the entire test.
  4. Increased productivity: Because FluentAssertions provides a more natural and expressive way of writing assertions, developers can write tests more quickly and with less effort.

In out test project

  1. Install FluentAssertions into the test project as NuGet package.
  2. Then we can use the Should() method to assert the test

Difference between Assert and FluentAssertions

  • Equal
    • Assert.Equal(expected, actual);
    • actual.Should().Be(expected);
  • Not Equal
    • Assert.NotEqual(expected, actual);
    • actual.Should().NotBe(expected);
  • Null
    • Assert.Null(actual);
    • actual.Should().BeNull();
  • Not Null
    • Assert.NotNull(actual);
    • actual.Should().NotBeNull();
  • True
    • Assert.True(actual);
    • actual.Should().BeTrue();
  • False
    • Assert.False(actual);
    • actual.Should().BeFalse();
  • Empty
    • Assert.Empty(actual);
    • actual.Should().BeEmpty();
  • Not Empty
    • Assert.NotEmpty(actual);
    • actual.Should().NotBeEmpty();
  • Null or empty
    • Assert.True(string.IsNullOrEmpty(actual)); Assert.True(actual == null || actual.Length == 0);
    • actual.Should().BeNullOrEmpty();
  • Should not be null or empty
    • Assert.False(string.IsNullOrEmpty(actual)); Assert.False(actual == null || actual.Length == 0);
    • actual.Should().NotBeNullOrEmpty();
  • Number should be positive
    • Assert.True(actual > 0);
    • actual.Should().BePositive();
  • Number should be negative
    • Assert.True(actual < 0);
    • actual.Should().BeNegative();
  • Number should be >= expected
    • Assert.True(actual >= expected);
    • actual.Should().BeGreaterThenOrEqualTo(expected);
  • Number should be <= expected
    • Assert.True(actual < expected);
    • actual.Should().BeLessThanOrEqualTo(expected);
  • Number should be in given range
    • Assert.True(actual >= minumum && actual <= maximum);
    • actual.Should().BeInRange(minimum, maximum);
  • Number should not be in given range
    • Assert.True(actual < minimum || actual > maximum);
    • actual.Should().NotBeInRange(minimum, maximum);
  • Check data type
    • Assert.IsType<ExcectedType>(actual);
    • actual.Should().BeOfType<ExpectedType>(); actual.Should().BeAssignableTo<ExpectedType>();
  • Compare properties of two objects
    • Assert.Equal(expected, actual); equals method should be overriden
    • actual.Should().BeEquivalentTo(expected); equals method need not be overriden
  • Compare properties of two objects
    • Assert.NotEqual(expected, actual); should not be equal and equals method should be overridden
    • actual.Should().BeNotEquivalentTo(expected); should not be equal and equals method need not be overridden

FluentAssertions - Collections

  • Empty or not
    • actualCollection.Should().BeEmpty();
    • actualCollection.Should().NotBeEmpty();
  • Have count or not
    • actualCollection.Should().HaveCount(expectedCount);
    • actualCollection.Should().NotHaveCount(expectedCount);
  • Have count greater or less euqla than
    • actualCollection.Should().HaveCountGreaterThanOrEqualTo(expectedCount);
    • actualCollection.Should().HaveCountLessThanOrEqualTo(expectedCount);
  • Have same count or not
    • actualCollection.Should().HaveSameCount(expectedCollection);
    • actualCollection.Should().NotHaveSameCount(expectedCollection);
  • Are those collections equivalent or not
    • actualCollection.Should().BeEquivalentTo(expectedColletion);
    • actualCollection.Should().NotBeEquivalentTo(expectedCollection);
  • Do those collection have the elements in same order or not
    • actualCollection.Should().ContainInOrder(expectedCollection);
    • actualCollection.Should().NotContainInOrder(expectedCollection);
  • Check if there is duplicated item
    • actualCollection.Should().OnlyHaveUniqueItems(expectedCount);
    • actualCollection.Should().OnlyContain(temp => condition);
  • Assert the collection is ascending or descending
    • actualCollection.Should().BeInAscendingOrder(temp => temp.Property);
    • actualCollection.Should().BeInDescendingOrder(temp => temp.Property);
  • Assert the collection is not ascending or descending
    • actualCollection.Should().NotBeInAscendingOrder(temp => temp.Property);
    • actualCollection.Should().NotBeInDescendingOrder(temp => temp.Property);

FluentAssertions - Exceptions

  • No asynchronous
    • delegateObj.Should().Throw<ExceptionType>();
    • delegateObj.Should().NotThrow<ExceptionType>();
  • Asynchronous
    • await delegateObj.Should().Throw<ExceptionType>();
    • await delegateObj.Should().NotThrow<ExceptionType>();
